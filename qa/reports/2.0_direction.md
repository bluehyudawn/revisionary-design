## 옵션 A — 디자인 시스템 우선 (Design System First)

### 핵심 전략
현재 4개 UI 구조는 유지하되, **공통 디자인 토큰·컴포넌트 라이브러리**를 먼저 확립하고 모든 화면을 이 시스템 위로 마이그레이션.

### 구체적 작업
1. Figma 토큰화: 컬러/타이포/스페이싱/라운드/섀도우 → 단일 source of truth
2. 공통 컴포넌트: Button(4종 × 3상태), Input, Select, Modal, Card, Badge, Tabs, Sidebar, TopBar
3. 기존 화면 리페이싱 (IA 변경 없이)
4. 개발자용 placeholder 제거, 브랜드 카피 정합
5. 온보딩 모달 버그 수정

### 장점
- 사용자 행동 영향 최소 (기존 워크플로 그대로)
- 엔지니어링 리스크 낮음 (시각·스타일 교체만)
- 빠르게 일관성 회복 가능 (6~10주 추정)
- 이후 어떤 방향으로 가든 **기반이 됨**

### 단점
- 본질적 문제(4개 UI 시스템·중복 기능)는 남음
- "리뉴얼" 체감은 약함 (내부는 완전히 정돈되나 대외적으로는 같아 보임)
- 사이드바 구조나 Create Exam 이중화 같은 **IA 레벨 이슈는 미해결**

### 추천 대상
- **현실적으로 가장 먼저 해야 하는 공통 작업**. B/C 옵션 전의 전제조건.

## 옵션 B — 역할 통합 (Role Consolidation, 2-persona)

### 핵심 전략
**educator (writer/teacher/manager 통합)** + **learner (student)** 두 persona 로 경량화. 권한 차이는 사이드바 항목 · 화면 내 액션 가시성으로 표현.

### 구체적 작업
1. Option A 의 디자인 시스템 선제 확립 (공통 전제)
2. 사이드바 통합: educator 공통 섹션 + 권한에 따라 User/Content Management 항목 노출
3. Create Exam 일원화: writer 의 간단 폼 + teacher/manager 의 리치 필드 → **한 화면, 필드 권한별 제어**
4. `/teacher/...` 경로 제거 → `/exam/create`, `/classroom`, `/question-bank` 등 의미 기반 라우트
5. Student 쪽은 현재 구조(Score, Billing, Subjects) 유지하되 디자인 정합
6. Manager 의 "Mock Exam" 관련 메뉴들을 "Exam → Mock" 으로 축약

### 장점
- IA 복잡도 대폭 감소 (13+ 메뉴 → 6~8 메뉴)
- 동일 팀(학교)의 writer/teacher 가 같은 화면 공유 → 교육/온보딩 비용 감소
- "역할별 다른 앱" 문제 해소
- Figma 스펙도 1회 디자인 후 권한만 토글

### 단점
- IA 변경이 커서 **기존 유저 재학습 필요**
- 권한 모델 재설계 필요 (백엔드 작업)
- 일부 manager 기능(User Management 등)은 기존 위치와 달라져 초기 혼란

### 추천 대상
- **브랜드가 일관된 "교육 제작 협업 도구"로 포지셔닝하고 싶다면** 이 방향. 1개 SaaS 로서의 정체성 확립.

## 옵션 C — 제품 재정의 (Progressive Disclosure Rewrite)

### 핵심 전략
UI 시스템을 1개로 완전 통합하고, **"내가 지금 할 일"** 중심의 태스크 기반 네비게이션으로 재설계. 역할은 대시보드 위젯·진입점 차이로만 표현.

### 구체적 작업
1. 옵션 A + B 전부 포함
2. 공통 Home: "오늘 할 일" 큐 (채점 N건, 작성 중 드래프트 N건, 새 시험 N건 등)
3. Global search / command palette (cmd+K)
4. Role persona 제거 — user attributes(can_create_exam, can_mark 등)로 전환
5. Mobile 디자인 본격 착수 (현재는 데스크톱 전용 추정)
6. 실제 결제 · 구독 모델 결정 (Payment 기능 실제 구축)

### 장점
- 가장 모던한 SaaS 경험 — 2.0 런칭 모멘텀 극대화
- 향후 확장(반/학교 단위 관리, AI 채점 등) 시 기반이 탄탄
- "revisionaryonline 은 하나의 교육 운영 OS" 내러티브 가능

### 단점
- 기간·비용 가장 큼 (3~6개월 추정)
- 백엔드 권한 모델·라우팅·데이터 구조 대대적 변경
- 이행 중에는 **기존 유저의 업무가 혼란** — 단계별 migration 플랜 필수
- 교육 관리자(기존 manager) 들의 learned workflow 를 재학습시켜야 함

### 추천 대상
- **투자 유치·시리즈 라운드 직전이거나, 향후 2년 제품 확장 로드맵이 명확할 때**.

## 공통 — 옵션 선택 무관하게 지금 당장 해야 하는 것 (P0 Hotfix)

릴리즈 전 또는 1~2주 내 조치 권장.

1. **브랜드 표기 통일** (Revisionary Online / RevisionaryOnline / revisiononline — 1개 선택 후 로고 / 이메일 도메인 / 문서 전체 적용)
2. **Teacher Welcome Modal**: 상태 영속화 + Step 2 Loading 버그
3. **Writer Create Exam**: eager validation 제거, Exam Specifications 필드 렌더 누락 수정
4. **Save Changes 초록 버튼** (student Subjects) → 보라 Primary 통합
5. **Payment Management placeholder** 개발 메모 제거
6. **Manager Focus color** 시안 → 보라
7. **`/teacher/exam/create`** URL 을 manager 도 쓰는 문제 — 최소한 어느 한 역할을 기본으로 정리

## 의사결정 체크리스트 (Alex용)

> 아래 질문에 답해주시면 A/B/C 중 어떤 옵션이 맞는지 좁힐 수 있습니다.

1. **타임라인**: 2.0 를 언제까지 시장에 내보내야 하나요? (3개월 / 6개월 / 1년)
2. **엔지니어링 여력**: 프런트/백엔드 각각 몇 명이 risk 받을 수 있나요?
3. **유저 영향**: 현재 유료/핵심 고객이 있다면, 재학습을 감수하는 변경이 가능한가요?
4. **포지셔닝**: 2.0 이후 이 제품은 **"시험지 제작 툴"** / **"교실 운영 플랫폼"** / **"학생 학습 관리"** 중 어디에 가장 가까워야 하나요?
5. **모바일**: 학생 쪽은 모바일 사용이 많을 텐데, 모바일 디자인 포함 여부?
6. **결제**: Payment 기능은 2.0 에 포함인가요, 아니면 일단 placeholder 유지?

## 제가 추천하는 경로

**단계적 실행: A → B 로 연결**

- **Phase 1 (0~8주)**: 옵션 A 전면 수행. 공통 Hotfix(P0) 포함. 디자인 시스템 확립.
- **Phase 2 (8~16주)**: 옵션 B 수행. educator 통합 + IA 정리 + `/teacher/...` 제거.
- **Phase 3 (옵션)**: 옵션 C 요소를 선택적으로 (cmd+K, 태스크 큐, 모바일) — 투자·시장 신호에 따라 결정.

**이유**: 옵션 C 를 바로 시작하면 현재의 일관성 문제를 해결하지 못한 채 재설계가 덮이게 되어 **"또 다른 새로운 UI 시스템"이 되는 리스크**가 큽니다. A 로 기반을 세우고 B 로 구조를 정리한 뒤, 필요하다면 C 의 개별 요소를 차후 흡수하는 것이 안전합니다.
